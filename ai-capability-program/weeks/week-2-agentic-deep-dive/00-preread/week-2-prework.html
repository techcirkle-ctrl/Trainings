Section 1 of 5~15 min

The agent loop

Why this matters to you

Last week you decided which part of your problem needs an agent. This week you build it — so you need to see what an agent actually is underneath the word. It's a loop, and the loop is where both the power and the risk live. Understanding it is what lets you design one and govern it.

↩ From Week 1: you learned an agent = LLM + tools + a loop — it takes a goal (not a question) and cycles think → act → observe → decide until it's done. That one line is the seed. Now we open the loop up and watch it run.

A chatbot answers once and waits. An agent is handed a goal and then loops: it thinks about what to do next, acts by calling a tool, observes the result, and decides whether it's done or needs another pass. Step it through and count the loops:

Loops: 0

Notice three things you'll own as a leader: it kept going on its own (autonomy), it only touched the real world through tools, and it stopped — handing the two it wasn't allowed to decide to a human. Those are the dials you set.

Leader's read

More loops and more tools = more autonomy = more power and more places to go wrong. An agent without a clear stopping rule and a human checkpoint isn't powerful — it's ungoverned. You own those two.

Section 2 of 5~15 min

Workflow vs agent — pick the least autonomy

Why this matters to you

Most things people proudly call "agents" are actually workflows — and that's good news: they're cheaper, more reliable, and far easier to govern. The single most valuable judgement you'll make this week is choosing the least autonomy that still solves the problem.

↩ From Week 1: "the cheapest layer that works wins." This week it sharpens to one level deeper: the least autonomy that works wins.

There's a spectrum. At one end, workflows — you fix the path the AI follows step by step. At the other, agents — the model decides the path itself, as you just saw. The named "architectures" are points along it. Pick a real task and watch where it lands:

Section 3 of 5~15 min

Tools — how an agent acts

Why this matters to you

An agent with no tools is just a chatbot talking to itself. Tools are how it reaches into your real systems — your CRM, a database, email. That makes them the most useful part and the riskiest: tools are exactly where the agent touches reality, so they're where your controls belong.

Here's the part that surprises people: the model never acts directly. When it needs a tool, it just writes a request — and your system decides whether and how to run it. Step through one tool call:

 
Press Step → to watch a single tool call, start to finish.
The model is mid-loop and needs something it can't answer from memory.

The key control point: the model only ever asks. Something you own actually runs the tool — so you decide what it may touch, what needs a human's nod first, and what data is even allowed near it.

One agent, many tools: MCP

Every tool has to be wired in. MCP (Model Context Protocol) is a standard plug — think USB-C — so any tool connects to any agent the same way. Wire a tool once; any agent can use it.

Section 4 of 5~15 min

Memory & state

Why this matters to you

Your agent forgets everything between sessions unless you design memory in. Deciding what it should remember versus look up is a leadership call — it drives cost, consistency, and what private data sits around.

↩ From Week 1: the context window = the desk. The model can only "see" what's on the desk right now. Everything you've learned about the desk applies here — we're just adding what happens when the desk fills, and where things go to persist.

Two kinds of memory. Short-term is the desk — this conversation, in view now, but wiped when you start a new chat and pushed off the edge when it overflows. Long-term is the filing cabinet — Project knowledge you save once, and it's there every time. Drive it:

Don't pour everything into memory. Decide what must persist (save it) versus what it can look up when needed. In no-code terms, Claude Projects is your persistent memory — and the more it remembers, the more you must govern what's in there.

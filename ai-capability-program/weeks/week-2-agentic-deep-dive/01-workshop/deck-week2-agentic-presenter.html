## 0. What an agent actually is (the fundamentals)

- **Agent = a GOAL + a LOOP + TOOLS + a STOPPING RULE.** Handed a goal (not a question), it loops — think -> act -> observe -> decide — calling tools, until a stopping rule says finish. Reacting once with text is a chatbot, not an agent.
- **Workflow vs agent is a spectrum.** In a workflow YOU fix the path in advance (prompt chaining, routing, parallelisation). In an agent the MODEL picks the path at run time (ReAct, Plan-and-Execute, Reflection, Multi-agent). The governing rule: **the least autonomy that works wins.** Start as far toward "workflow" as the problem allows.
- **Tools: the model only ASKS.** When an agent "uses a tool" the model writes a structured request; YOUR system runs it and returns the result as an observation. That separation is the control point — you decide what it may touch and what needs approval.
- **MCP** is a standard plug (USB-C for tools): wire a tool once, any agent can use it. In no-code terms this is Projects -> Connectors.
- **Memory: short-term vs long-term.** Short-term is the context window (this chat — wiped on a new chat, overflows when full). Long-term is Project knowledge (saved once, persists every run). Decide what must PERSIST vs be LOOKED UP.

## 1. What makes a GOOD defined agent (the bar for v1)

A good v1 defined agent is:
- **Least autonomy that works.** Mostly a fixed path, with the smallest genuinely agent-shaped core. If a workflow does the job, build the workflow.
- **Bounded.** One job, one workflow — not a department transformation. Buildable as a v1 in a session.
- **Grounded.** It has the knowledge it needs (policies, house style, criteria for a good output) in Project knowledge.
- **Safe to let loose.** It has instructions (role · task · guardrails) AND a human checkpoint on any consequential action. Tools-on-checkpoint-off is the dangerous state.
- **Stoppable.** A clear stopping rule — how it knows it's done, and what it escalates.

A WEAK v1 is: over-built (agent where a workflow would do), unbounded, ungrounded, or acting with no checkpoint.

## 2. The parts of an agent (assemble these)

- **Instructions (required)** — role, task, guardrails (what it must never do).
- **Human checkpoint (required)** — the step where it must stop and ask a person before a consequential action.
- **Knowledge (recommended)** — the documents/policies it needs, in Project knowledge.
- **Tools / connectors (recommended)** — how it acts; the fewest that let it do the job.
- **Stopping rule (recommended)** — how it knows it's done or must escalate.

## 3. Defined-agent spec sheet (use if the leader needs a problem, or as probes)

Each is a real, bounded, mostly-workflow-with-a-small-agent-core job for this GCC's world:

- **Engineering Change Request (ECR) Reviewer.** Read an incoming ECR, classify it, check it against the relevant spec/standard, flag risk, draft the first response. *Least autonomy: mostly routing + RAG over past ECRs; agent-shaped only if it files/routes in a system. Checkpoint: human approves the risk flag and any routing.*
- **Customer Escalation Triage.** Read a support escalation, pull the relevant product docs, draft a diagnosis and next step, route by severity. *RAG-heavy; agent-shaped if it opens/updates the ticket. Checkpoint: human approves anything customer-facing.*
- **Meeting Decision Tracker.** From meeting notes, extract decisions, owners and deadlines; chase the open ones. *LLM to extract; agent-shaped only if it actually sends the chase messages and loops. Checkpoint: human approves outbound chase messages.*
- **Vendor Proposal Analyser.** Normalise differently-structured vendor proposals to a like-for-like comparison, flag hidden terms and risks against a standard. *Mostly LLM + RAG on the standard; rarely agent-shaped. Checkpoint: human signs off the recommendation.*

## 4. Data sensitivity — at the tool boundary

Tools touch real systems, so "what data may go near this?" is a tool question. Name the data types that must NOT go into an AI tool (supplier contracts/pricing, customer/employee PII, unpublished financials, source code, anything under NDA) and keep them in the sanctioned workspace account. The rule isn't "don't use internal data" — it's "use it only in the sanctioned tool, and put a human where the action has consequences."

## 5. Human-in-the-loop — where authority runs out

For any action with consequences (external email, committing spend, a customer promise, changing a record), name the step where a human must approve before the agent acts. The more consequential the action, the earlier the checkpoint. That approval point is the governance the leader designs in.
